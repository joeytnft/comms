# GatherSafe Security Model

This document describes what GatherSafe's encryption and access controls actually do — and what they don't. It supersedes any earlier "zero-knowledge" or "Signal Protocol / E2E" claims that appeared in the README, CLAUDE.md, or product copy.

## Summary

GatherSafe is a **server-managed-keys** application with strict per-org / per-group authorization, biometric/PIN device lock, and TLS in transit. It is **not** zero-knowledge. The server stores group keys, can read message content if the database is compromised, and is in scope for legal discovery.

## What protects user data

| Layer | Mechanism | Threat addressed |
|---|---|---|
| In transit | TLS 1.2+ on every API and Socket.IO connection | Network eavesdropping, MITM |
| At rest (DB) | Postgres at-rest encryption (hosting provider) + AES-256-GCM application-layer envelope encryption for message bodies and incident details | Disk theft, backup exfiltration without keys |
| At rest (object storage) | Signed, short-TTL URLs for PTT recordings and incident photos. Bucket is private; URLs expire | URL guessing, scraping, indefinite access by removed members |
| On device | iOS Keychain / Android Keystore for tokens; MMKV for non-sensitive caches; biometric or PIN app lock | Lost / stolen phone |
| Authorization | Per-org JWT, per-group membership checks on every endpoint and socket event, role-based admin checks for destructive ops, lead → sub-group hierarchy enforced on read paths | Cross-tenant data access, privilege escalation, IDOR |
| Audit | Sensitive ops (panic alert, incident create, member kick, role change, alert delete) are logged to an append-only audit table with actor, target, and timestamp | Insider abuse review |

## What does NOT protect user data

- **Server compromise.** A leaked database backup, a SQL injection, or a malicious admin with DB access can read all message content. Group keys are stored server-side.
- **Compromise of an org admin account.** Org admins can mint additional admins, invite users, and read group content. There is no second factor for admin operations today.
- **Legal process.** Subpoenas, civil discovery, and law enforcement requests can compel access to message content and recordings.
- **Hosting-provider trust.** Railway, Supabase, Expo (push), Apple, Google, RevenueCat, LiveKit, and the email provider all see at least metadata. Push notifications include the alert title and triggering user's name (not the full message body).

## Key management

- **Group keys** are 32-byte AES keys generated by the server with `crypto.randomBytes(32)` on first request, stored in `groups.groupKey`, and served over TLS to authenticated members of the group.
- **Key rotation** happens automatically when a member is removed from a group. The new key is generated server-side and distributed to remaining members on their next message-fetch. Older messages remain readable to former members who cached the previous key — this is operationally documented but not cryptographically prevented.
- **Voice (LiveKit)** uses SRTP between client and SFU. No application-layer E2EE is layered on top. Recordings are stored encrypted at rest in private object storage and accessed via signed URLs.

## Threat scenarios

| Scenario | Outcome |
|---|---|
| Lost / stolen unlocked phone | Attacker has session-bound access until tokens expire (15 min access, 7 day refresh) and can use the app. Mitigated by biometric/PIN app-lock. |
| Lost / stolen locked phone | Attacker has nothing; secrets are in Keychain/Keystore, app requires biometric/PIN. |
| Database backup exfiltration | All ciphertext + group keys leaked. Effectively all messages decryptable. **Highest-priority risk.** |
| Network MITM | TLS prevents passive interception. Pinning is **not** implemented today. |
| Compromised org admin account | Full access to org data. No second factor for admin ops. |
| Removed group member | Loses access to future messages on key rotation. Cached old-key messages remain readable. |
| Subpoena | Server can produce decrypted content. |

## Out of scope for this model

- Zero-knowledge / true E2E encryption (Signal Protocol, MLS). Documented as a possible future direction; not implemented and not promised.
- Forward secrecy at the application layer.
- Multi-device key sync.
- Cryptographic verification of identity (safety numbers).
